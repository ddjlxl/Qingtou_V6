## 🎯 硬性规则（必须遵守）

### 1.1 代码提交规则
1. **禁止自动提交**：严禁AI自动提交代码
2. **提交信息规范**：必须使用详细中文描述
3. **操作范围锁定**：严格限制在 `E:\Qingtou_V6` 目录内

### 1.2 测试执行规则
1. **禁止自动运行测试**：代码完成后静默报告，仅用户要求"验证"时才执行
2. **测试覆盖率**：核心功能必须达到80%以上测试覆盖率

### 1.3 文档更新规则
1. **代码变更必须更新文档**：API变更、数据库变更必须同步更新文档
2. **文档版本控制**：所有文档必须有版本号和更新日期

## 🚫 绝对禁止的行为

### 2.1 代码质量相关
> 代码质量红线详见 [ai-constraints.md](ai-constraints.md)，此处不重复。核心：禁止 `any`、禁止 `console.log`、文件 ≤300 行、函数 ≤50 行。

### 2.2 架构设计相关
- ❌ 禁止在Store中混入Mock数据和API逻辑
- ❌ 禁止在组件中直接操作DOM
- ❌ 禁止硬编码敏感信息（密码、密钥等）

### 2.3 业务逻辑相关
- ❌ 禁止在前端进行核心业务计算（如运费计算）
- ❌ 禁止绕过权限验证
- ❌ 禁止修改需求文档中未定义的功能

## ✅ 必须遵守的规范

### 3.1 TypeScript规范
- ✅ 所有变量和函数必须有明确的类型定义
- ✅ 接口优先于类型别名
- ✅ 使用枚举代替魔法字符串

### 3.2 Vue组件规范
- ✅ 使用 `<script setup>` 语法
- ✅ Props和Emits必须有类型定义
- ✅ 组件名称使用PascalCase

### 3.3 文件组织规范
- ✅ 按功能模块组织目录结构
- ✅ 每个模块必须有清晰的职责划分
- ✅ 公共组件和工具函数统一管理

## 🔄 开发流程规范

### 5.1 功能开发流程
```
1. 阅读需求文档 → 2. 设计接口 → 3. 编写测试 → 4. 实现功能 → 5. 代码审查
```

### 5.2 数据库变更流程
```
1. 更新数据库设计文档 → 2. 编写迁移脚本 → 3. 测试数据操作 → 4. 部署验证
```

### 5.3 问题修复流程
```
1. 重现问题 → 2. 分析原因 → 3. 编写修复 → 4. 验证修复 → 5. 更新文档
```

## 📊 质量指标要求

### 6.1 代码质量指标
> 详见 [ai-constraints.md](ai-constraints.md) 第六章质量指标要求。
- **类型安全**：`any` 类型使用率 0%
- **函数复杂度**：单个函数不超过50行
- **文件复杂度**：单个文件不超过300行
- **测试覆盖率**：核心功能 > 80%

### 6.2 性能指标
- **页面加载**：首屏加载时间 < 3秒
- **API响应**：关键接口响应时间 < 2秒
- **内存使用**：无内存泄漏

## 🆘 遇到问题的处理流程

### 7.1 技术问题
1. 先查阅相关文档
2. 分析V4中的类似实现
3. 如果无法解决，向用户说明问题并提供解决方案建议

### 7.2 需求不明确
1. 暂停开发
2. 向用户确认需求细节
3. 更新需求文档后再继续

### 7.3 性能问题
1. 分析性能瓶颈
2. 提供优化方案
3. 获得用户同意后实施优化
