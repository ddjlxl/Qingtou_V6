# V6 Skills使用指南

> **版本**：v3.0  
> **用途**：指导如何使用V6 Skills体系  
> **适用对象**：V6项目开发者

---

## 📖 概述

V6 Skills是一套完整的AI编程工作流体系，基于SpecForge V3方法论设计，包含：

- **3个核心机制**：边界守卫、项目上下文协议、AI约束机制
- **9个项目级Skills**：需求澄清、产品概述、技术栈选型、目录结构、开发规范、路线图规划、项目初始化、功能需求澄清、功能技术设计
- **2个功能级Skills**：任务规划、功能实现（统一TDD执行器）
- **5个通用级Skills**：测试、BUG修复、代码审查、性能优化、功能迭代

---

## 🎯 核心理念

### 1. 系统化流程
从模糊的想法到可执行的代码，每一步都有明确的指引。

### 2. 强制性约束
边界守卫确保AI在正确的时间做正确的事。

### 3. 自动化上下文
项目上下文协议确保AI对项目有完整认知。

### 4. 零容忍质量
AI约束机制确保代码质量，避免技术债务。

### 5. 垂直切片开发
编码实现层统一为一个Skill，穿透所有技术层（数据库 → API → Store → UI）一口气完成，上下文连贯不丢失。

## 📚 完整工作流

### 十一阶段开发流程

```
项目需求澄清 → 产品规划 → 项目技术设计 → 路线图规划 → 项目初始化
                                                          ↓
功能需求澄清 → 功能技术设计 → 任务规划 → 编码实现 → 测试验证
                                              ↑           ↓
                                              └── 功能迭代 ←┘
```

## ⚙️ 核心机制说明

### 边界守卫（Guardrails）

**作用**：强制AI在正确的阶段做正确的事

**十一阶段边界**：
1. 项目需求澄清阶段 - 只能提问和讨论
2. 产品规划阶段 - 只能审查和建议
3. 项目技术设计阶段 - 只能设计方案
4. 路线图规划阶段 - 只能规划排序
5. 项目初始化阶段 - 只能创建骨架
6. 功能需求澄清阶段 - 只能澄清功能需求
7. 功能技术设计阶段 - 只能设计技术方案
8. 任务规划阶段 - 只能拆分任务
9. 编码实现阶段 - 可以写代码（统一TDD执行器）
10. 测试验证阶段 - 可以测试和修复
11. 功能迭代阶段 - 可以变更管理

---

### 项目上下文协议（Project-Context）

**作用**：确保AI对项目有完整认知

**必读文档**：
- V6需求文档
- V6数据库设计
- V6开发规则
- V4踩坑记录

---

### AI约束机制（AI-Constraints）

**作用**：确保代码质量，避免技术债务

**零容忍规则**：
- 禁止使用 `any` 类型
- 禁止遗留 `console.log`
- 禁止单个文件超过300行
- 禁止函数超过50行

## 💡 最佳实践

### 1. 按流程使用
不要跳过步骤，按照十一阶段流程一步步执行。

### 2. 认真对待澄清
需求澄清阶段是整个流程的基石，不要急于进入设计阶段。

### 3. 利用垂直切片
编码实现时，一个切片内从数据库到UI一口气做完，AI不会丢失上下文。

### 4. 及时更新文档
当需求或设计变更时，及时更新文档（使用 `feature-iteration` 的 CR 编号机制）。

### 5. 参考V4问题
开发时参考V4踩坑记录，避免重复错误。

### 6. 持续优化Skills
根据实际使用情况，持续优化Skills。

## ❓ 常见问题

### Q1：为什么需要Skills？
**A**：Skills提供系统化的流程，确保AI在正确的约束下工作，避免跑偏和重复错误。

### Q2：必须按照流程使用吗？
**A**：是的，十一阶段流程是经过精心设计的，跳过步骤容易出问题。

### Q3：为什么编码实现合并为一个Skill？
**A**：垂直切片理念——一个用户行为涉及数据库、API、Store、UI多层，分开做会丢失上下文。统一执行器穿透所有层，一口气完成。

### Q4：可以自定义Skills吗？
**A**：可以！根据项目特点创建专属的Skills。

### Q5：如何知道使用哪个Skill？
**A**：参考"Skills使用场景"部分，根据当前任务选择合适的Skill。

### Q6：功能迭代和BUG修复有什么区别？
**A**：
- **BUG修复**：代码行为与AC不一致，走TDD修复流程（含问题分级P0-P3）
- **功能迭代**：用户想要的行为与AC不同，走完整的影响分析→更新设计→重新规划→TDD实现流程（含CR编号）
