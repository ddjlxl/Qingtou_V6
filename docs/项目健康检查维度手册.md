# V6 项目健康检查维度手册

> **版本**：v1.0
> **创建日期**：2026-05-21
> **用途**：供 AI Agent 定期执行项目健康扫描，每个维度包含检查方法、判定标准和修复指引
> **执行频率**：建议每周一次全量扫描，每次功能迭代后做增量扫描

---

## 使用说明

1. AI Agent 收到"检查项目健康"指令时，按本文档逐维度执行
2. 每个维度输出：通过 / 警告 / 失败，附具体文件和行号
3. 最终输出汇总报告，按严重程度排序
4. 只报告能通过"三道筛子"的问题（会导致 Bug / 会导致线上故障 / 三个月后开发者会骂）

## 维度一：零容忍规则（9 条红线）

> 来源：V4 惨痛教训，违反任何一条必须立即修复

### 1.1 检查项

| # | 规则 | 检查命令/方法 | 判定标准 |
|---|------|-------------|---------|
| 1 | 禁止 `any` 类型 | `Grep` 搜索 `: any` 和 `as any`（排除 node_modules、.d.ts） | 出现即失败 |
| 2 | 禁止 `console.log` | `Grep` 搜索 `console.log`（排除 __mocks__、__tests__） | 出现即失败 |
| 3 | 文件不超过 300 行 | 遍历 `apps/frontend/src` 和 `apps/server/app` 下所有 .ts/.vue/.py 文件，统计行数 | 超过即失败 |
| 4 | 函数不超过 50 行 | `pnpm lint` 检查 `max-lines-per-function` 规则 | 报错即失败 |
| 5 | 组件必须处理三种状态 | 读取每个 .vue 组件，检查是否有 v-loading / 空状态 / 错误处理 | 缺失即警告 |
| 6 | 异步操作必须有 try-catch | 搜索 `async function` 和 `await`，检查是否有 try-catch 包裹 | 缺失即失败 |
| 7 | 样式必须 scoped | 搜索 `<style>` 标签，检查是否有 `scoped` 属性 | 无 scoped 即失败 |
| 8 | 模块间只能通过 index.ts 引用 | 搜索跨模块 import，检查是否引用了其他模块的内部路径 | 直接引用内部即失败 |
| 9 | Store 禁止混入 Mock 数据 | 读取 store 文件，检查是否有硬编码的测试数据或 mock fallback | 出现即失败 |

### 1.2 检查脚本示例

```bash
# 前端零容忍快速检查
cd apps/frontend
pnpm lint                    # ESLint 包含 max-lines、max-lines-per-function
pnpm type-check              # TypeScript 严格模式

# 后端零容忍快速检查
cd apps/server
python -m pytest tests/ -q   # 测试套件
```

### 1.3 AI Agent 补充检查（lint 无法覆盖的）

- 用 `Grep` 搜索 `: any` → 排除 `.d.ts` 和 `node_modules`
- 用 `Grep` 搜索 `console.log` → 排除 `__tests__` 和 `__mocks__`
- 用 `Grep` 搜索跨模块直接引用：`from '@/modules/(?!dispatch|driver|fleet|auth|shared)[^']*/(?!index)` 模式
- 逐个读取 store 文件，检查是否有 `if (mock)` 或 `fallbackToMock` 模式

## 维度二：类型安全

> V4 有 257 处 any，导致重构不敢动、运行时才暴露错误

### 2.1 检查项

| # | 检查点 | 方法 | 判定标准 |
|---|--------|------|---------|
| 1 | 前端 TypeScript 严格模式 | 读取 `tsconfig.json`，确认 `strict: true` | 非严格即失败 |
| 2 | 无 `as any` 类型断言 | `Grep` 搜索 `as any` | 出现即失败 |
| 3 | 无 `@ts-ignore` / `@ts-expect-error` | `Grep` 搜索 `@ts-ignore` 和 `@ts-expect-error` | 出现即警告（确认是否有正当理由） |
| 4 | API 响应类型完整 | 读取 service 层文件，检查每个 API 调用是否有返回类型标注 | 缺失即警告 |
| 5 | Pydantic schema 与 Model 字段对齐 | 对比 `app/schemas/` 和 `app/models/` 的字段定义 | 不一致即失败 |
| 6 | 前端类型与后端 schema 对齐 | 对比前端 types 和后端 Pydantic schema 的字段名和类型 | 不一致即警告 |

### 2.2 对齐检查方法

```
1. 读取 apps/server/app/schemas/ 下所有 schema 文件，提取字段名列表
2. 读取 apps/server/app/models/ 下对应 model 文件，提取字段名列表
3. 对比：schema 有但 model 没有 → 失败；model 有但 schema 没有 → 警告
4. 读取 apps/frontend/src/modules/*/types/ 下类型定义
5. 与后端 schema 对比：字段名不一致 → 警告
```

## 维度三：并发安全

> V6 已发现并修复了车辆分配的竞态条件，需持续监控

### 3.1 检查项

| # | 检查点 | 方法 | 判定标准 |
|---|--------|------|---------|
| 1 | 关键资源查询是否加行锁 | 搜索 `select(...).where(...)` 后是否跟 `.with_for_update()`，特别关注：库存扣减、状态变更、资源分配 | 关键路径缺失即失败 |
| 2 | 状态变更是否有条件检查 | 搜索 `update(...).values(status=...)` 前是否有 `where(status=old_status)` 条件 | 无条件更新即失败 |
| 3 | 多表操作是否在同一事务 | 搜索 `await db.commit()` 的位置，检查连续的多表操作是否在同一个 commit 前 | 跨事务即失败 |
| 4 | 是否有 TOCTOU 竞态 | 搜索"先查后改"模式：先 select 判断条件，再 update 执行变更，中间无行锁 | 存在即失败 |

### 3.2 高风险路径清单

以下业务路径必须加行锁，每次检查重点验证：

- 车辆分配（`assign_order`）：车辆状态检查 → 分配
- 订单完成（`complete_order`）：订单状态检查 → 完成
- 订单超时（`check_overdue_orders`）：订单状态检查 → 标记超时
- 司机状态变更：司机在线状态 → 分配

## 维度四：数据库健康

### 4.1 检查项

| # | 检查点 | 方法 | 判定标准 |
|---|--------|------|---------|
| 1 | N+1 查询 | 搜索 `select(Model)` 后是否对关联字段使用 `selectinload` / `joinedload` | 关联查询缺失即警告 |
| 2 | 迁移与模型同步 | 运行 `alembic check`（Alembic 1.10+），或对比 model 定义与实际数据库 | 不同步即失败 |
| 3 | 测试数据库 schema 同步 | 对比测试数据库表结构与 model 定义（`Base.metadata.create_all` 只创建不更新） | 不同步即失败 |
| 4 | 缺失索引 | 读取 model 文件，检查高频查询字段（外键、status、business_type）是否有 `index=True` | 缺失即警告 |
| 5 | 时区处理 | 搜索 `DateTime` 列定义，检查是否使用 `timezone=True`；搜索 `datetime.now()` 是否带 `timezone.utc` | 不一致即失败 |
| 6 | 大表无分页 | 搜索 `list_*` / `get_*_list` API，检查是否有 `limit` / `offset` 参数 | 无分页即失败 |

### 4.3 测试数据库同步检查方法

```
1. 连接测试数据库（qingtou_v6_test）
2. 查询 information_schema.columns 获取实际表结构
3. 读取 model 文件获取定义的列
4. 对比：model 有但数据库没有 → 失败（需要重建测试数据库或手动 ALTER）
5. 修复方法：DROP DATABASE qingtou_v6_test，让 conftest.py 在下次测试时重建
```

## 维度五：架构合规

> 模块化架构是 V6 的核心设计，违反会导致耦合和重构困难

### 5.1 检查项

| # | 检查点 | 方法 | 判定标准 |
|---|--------|------|---------|
| 1 | 模块间引用走 index.ts | 搜索 `from '@/modules/X/` 开头的 import，排除 `from '@/modules/X'` 和 `from '@/modules/X/index` | 直接引用内部即失败 |
| 2 | 模块 index.ts 导出完整 | 读取每个模块的 `index.ts`，对比模块内实际导出的类型/store/组件 | 缺失导出即警告 |
| 3 | 分层依赖方向正确 | 检查依赖链：Types → Config → Repo → Service → Runtime → UI，不允许反向依赖 | 反向依赖即失败 |
| 4 | shared 层无业务逻辑 | 读取 `shared/` 下文件，检查是否包含特定业务的类型或逻辑 | 出现即失败 |
| 5 | 后端 API 版本化 | 搜索路由注册，检查是否都在 `/api/v1/` 下 | 不在版本路径下即警告 |

### 5.2 依赖方向检查方法

```
允许的依赖方向：
  types/ → 无依赖
  config/ → 可依赖 types
  repositories/ → 可依赖 types, config
  services/ → 可依赖 types, config, repositories
  stores/ → 可依赖 types, services
  components/ → 可依赖 types, stores
  views/ → 可依赖 types, stores, components

禁止的依赖方向：
  types/ → 依赖任何其他层
  services/ → 依赖 stores 或 components
  repositories/ → 依赖 services
```

## 维度六：安全合规

### 6.1 检查项

| # | 检查点 | 方法 | 判定标准 |
|---|--------|------|---------|
| 1 | API 路由有认证保护 | 读取路由文件，检查每个端点是否有 `Depends(get_current_user)` 或在受保护的路由组下 | 无认证即失败 |
| 2 | 角色权限校验完整 | 搜索 `require_role` / `role_check`，对比业务需求中的角色矩阵 | 缺失即失败 |
| 3 | 无 SQL 注入风险 | 搜索 `text()` / `execute()` 中的字符串拼接 | 存在拼接即失败 |
| 4 | 无 XSS 风险 | 搜索 `v-html`，确认数据来源是否已消毒 | 未消毒即失败 |
| 5 | 敏感数据不暴露 | 搜索 API 响应 schema，检查是否包含 password / secret / token 字段 | 暴露即失败 |
| 6 | JWT 配置安全 | 读取 `config.py`，检查 SECRET_KEY 是否从环境变量读取、过期时间是否合理 | 硬编码即失败 |
| 7 | CORS 配置合理 | 读取 CORS 中间件配置，检查 `allow_origins` 是否为 `*` | 生产环境 `*` 即失败 |
| 8 | 文件上传安全 | 搜索上传相关代码，检查是否有文件类型/大小限制 | 无限制即失败 |

## 维度七：API 一致性

### 7.1 检查项

| # | 检查点 | 方法 | 判定标准 |
|---|--------|------|---------|
| 1 | 响应格式统一 | 读取 API 端点，检查是否都返回 `{ code, message, data }` 格式 | 不一致即警告 |
| 2 | HTTP 状态码正确 | 搜索 `JSONResponse` / `Response`，检查状态码是否语义正确（201 创建、404 不存在等） | 不正确即警告 |
| 3 | 错误处理统一 | 搜索异常处理，检查是否都通过 `AppException` / 全局异常处理器返回 | 直接返回非标格式即失败 |
| 4 | 分页参数统一 | 搜索分页 API，检查参数名是否统一（`page` / `page_size`） | 不统一即警告 |
| 5 | Schema 验证完整 | 读取 Pydantic schema，检查字段是否有 `Field` 约束（min_length、pattern 等） | 输入字段无约束即警告 |

## 维度八：前端状态管理

> V4 的 computed 副作用和状态不同步是高频 Bug 来源

### 8.1 检查项

| # | 检查点 | 方法 | 判定标准 |
|---|--------|------|---------|
| 1 | computed 无副作用 | 搜索 `computed(` 块，检查内部是否有 `Math.random()`、`Date.now()`、赋值操作、API 调用 | 存在即失败 |
| 2 | Store 操作后状态一致 | 读取 store actions，检查写操作（创建/更新/删除）后是否刷新关联状态 | 未刷新即失败 |
| 3 | 组件卸载时清理副作用 | 搜索 `onMounted` / `watch`，检查是否有对应的 `onUnmounted` 清理 | 未清理即警告 |
| 4 | 响应式数据正确使用 | 搜索 `ref` / `reactive` 赋值，检查是否丢失响应性（如解构 reactive） | 丢失即失败 |
| 5 | loading / error 状态管理 | 读取 store，检查每个 async action 是否正确管理 loading 和 error | 缺失即警告 |

## 维度九：测试健康

### 9.1 检查项

| # | 检查点 | 方法 | 判定标准 |
|---|--------|------|---------|
| 1 | 测试通过率 | 运行 `pnpm test`（前端）和 `pytest`（后端） | 有失败即失败 |
| 2 | 测试隔离性 | 读取测试文件，检查是否依赖其他测试的数据或执行顺序 | 依赖即失败 |
| 3 | 关键路径有测试 | 检查以下路径是否有测试覆盖：认证、订单 CRUD、车辆分配、状态流转 | 缺失即警告 |
| 4 | Mock 与实际接口对齐 | 读取 `__mocks__` 和 `__tests__` 中的 mock 数据，对比实际 API 响应类型 | 不一致即警告 |
| 5 | 无跳过的测试 | 搜索 `skip` / `xit` / `@pytest.mark.skip` | 存在即警告（需确认原因） |
| 6 | 测试数据库 schema 与模型同步 | 对比测试数据库和 model 定义 | 不同步即失败 |

## 维度十：死代码与依赖健康

### 10.1 检查项

| # | 检查点 | 方法 | 判定标准 |
|---|--------|------|---------|
| 1 | 无未使用的导出 | 读取模块 `index.ts` 导出的内容，搜索项目中是否有引用 | 无引用即警告 |

| 6 | 无未使用的依赖 | 运行 `pnpm dlx depcheck`（前端） | 未使用依赖即警告 |
