## P1 重要项（建议修复后再开发）

### 项目规范合规性

- [ ] **无 `any` 类型**：design.md 中所有 TypeScript 类型定义未使用 `any`
- [ ] **无 `console.log`**：design.md 中所有代码示例未使用 `console.log`，使用了日志系统
- [ ] **文件 ≤ 300 行**：tasks.md 规划的所有新建文件预估行数不超过 300 行（特别关注 `useFleetStore.ts`、`fleet.py`）
- [ ] **函数 ≤ 50 行**：design.md 中核心逻辑函数预估行数不超过 50 行
- [ ] **无推测性功能**：design.md 和 tasks.md 中未出现 requirements.md "本次不做"清单中的功能
- [ ] **外科手术式修改**：tasks.md 中每个任务只修改与对应 AC 直接相关的代码，无"顺手重构"

### TDD 合规性

- [ ] **测试任务存在**：每个实现任务（前端组件/后端 API）有对应的测试任务
- [ ] **测试任务引用 AC**：每个测试任务的 Description 或验证标准中引用了具体的 AC 编号
- [ ] **测试覆盖边界**：测试任务覆盖了 Happy Path 和 Edge Case（不只是正常流程）

### 核心逻辑正确性

- [ ] **车辆状态更新**：`update_vehicle_status` 的事务边界正确（dispatch 调用 → fleet 服务层 → 数据库提交）
- [ ] **证照预警计算**：`get_certificate_warning_count` 正确排除已过期证照（`expiry_date >= today`），包含 30 天内到期证照
- [ ] **运输流水导入**：逐行处理的事务策略明确（单行失败不影响其他行 vs 全部回滚），重复记录跳过逻辑正确
- [ ] **级联删除**：删除车辆/司机时，关联证照的数据库记录和物理附件文件同步清理，清理失败时的降级策略明确
- [ ] **司机关联冲突**：`bind_driver_to_vehicle` 两步确认流程中，第二步重新执行完整冲突检查（不依赖第一步缓存）
- [ ] **证照类型校验**：`validate_cert_type` 正确拒绝车辆证照类型用于司机（反之亦然）

### 任务依赖

- [ ] **无循环依赖**：tasks.md 依赖关系图中无 A → B → A 的循环
- [ ] **Depends On 指向存在**：每个任务的 Depends On 指向的任务编号在 tasks.md 中存在
- [ ] **关键路径正确**：依赖关系图的关键路径与任务 Depends On 链一致
- [ ] **可并行任务标注**：可并行开发的任务在依赖关系图中正确标注

### 业务规则映射

- [ ] **AC-027 映射**：调度中心分配任务 → 车辆状态变为"运输中"，design.md 有明确实现点
- [ ] **AC-028 映射**：任务超时 4 小时 → 车辆状态变为"超时"，design.md 明确了 dispatch 负责检查、fleet 负责更新
- [ ] **AC-029 映射**：每日自动检查证照有效期 → 预警计入统计，design.md 有定时任务实现
- [ ] **AC-030 映射**：司机关联冲突 → 提示确认 → 清空原关联，design.md 有两步确认流程
- [ ] **AC-031 映射**：调度员和管理员权限相同，design.md 有路由守卫和 API 权限校验

### 范围界定

- [ ] **"本次不做"未泄露**：requirements.md "本次不做"清单中的功能在 design.md 和 tasks.md 中均未出现
- [ ] **依赖模块状态正确**：requirements.md 标注的依赖模块（auth、shared、database-model）状态（已完成 ✅）与实际一致
